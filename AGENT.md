# AGENT.md

このファイルは、AI エージェントが AI Workbench で作業するときの最小ガードレールです。
詳細な要件、設計、品質運用をここに複製せず、正本となる Issue、`docs/`、README を参照してください。

## 1. リポジトリの目的

- AI Workbench は、AI を活用した知的生産・開発支援ツール群を育てるモノレポです。
- 最初のアプリは `apps/prompt-trail` に配置する PromptTrail です。
- PromptTrail は Prompt、Context、Recipe、Run、Link を作業資産として扱います。
- AI への依頼、Chat、Issue、PR、成果物までの Trail を追跡できる状態を目指します。
- 日常コマンドの入口は root `README.md` です。
- 要件、ロードマップ、設計判断は `docs/` 配下を正本とします。
- 品質ゲートと開発運用は `docs/development/quality-gates.md` を正本とします。

## 2. 作業開始時の参照順序

作業前に、少なくとも次の順序で確認してください。

1. 対象 GitHub Issue。
2. 対象 Issue の親 Issue、関連 Issue、リンクされた議論。
3. 対象領域に対応する `docs/product/` または `docs/architecture/`。
4. 品質確認の正本である `docs/development/quality-gates.md`。
5. 変更対象モジュールの既存実装、テスト、README、近接するドキュメント。

Issue に明記された対象範囲と非対象範囲を、作業計画と最終報告の基準にしてください。

## 3. 正本と変更境界

- 作業単位の要求、対象範囲、非対象範囲は GitHub Issue を正本とします。
- 要件、設計、品質運用は既存の `docs/` を正本とし、このファイルに詳細仕様を複製しません。
- README は日常的なセットアップ、コマンド、ドキュメント入口として扱います。
- Issue にない依存追加、依存更新、lockfile 更新を混在させないでください。
- Issue にない大規模リファクタリング、ファイル移動、責務変更を混在させないでください。
- Issue にない仕様変更、UI 方針変更、データモデル変更を混在させないでください。
- 設計資料と実装の差異を見つけた場合は、独自に仕様を変更しないでください。
- 差異、判断、保留事項は Issue または PR に明記してください。
- 小さな修正でも、変更目的に必要な最小範囲へ留めてください。

## 4. 実装上の共通ルール

- pnpm Workspace を使います。
- 依存取得は root で `pnpm install --frozen-lockfile` を実行することを標準とします。
- `pnpm-lock.yaml` は、依存追加または依存更新が必要な変更に限って更新します。
- npm や yarn の lockfile を追加しないでください。
- PromptTrail では domain、DB、Repository、UI の責務境界を守ってください。
- UI から Dexie または IndexedDB を直接操作しないでください。
- DB アクセスは、後続で整備されるアクセス層や Repository 境界を経由する前提で扱ってください。
- テスト、UI、設計資料、運用文書の責務を混ぜないでください。
- 変更対象に既存テストがある場合は、利用者から見える振る舞いを優先して確認してください。
- CSS class、内部配列順、実装都合だけに依存するテスト追加は避けてください。

## 5. 品質確認

- 変更種別に応じたローカル確認は `docs/development/quality-gates.md` を正本とします。
- 文書のみの変更でも、原則として Markdown 整形確認を実行してください。
- 標準の品質確認コマンドは次のとおりです。
  - `pnpm lint`
  - `pnpm format:check`
  - `pnpm test`
  - `pnpm test:e2e`
  - `pnpm build`
- PromptTrail だけを対象にする場合は `pnpm --filter prompt-trail ...` を使います。
- PR CI を最終確認とします。
- CI 失敗時に、CI 上で自動整形や仕様変更を行わないでください。
- 失敗したコマンド、原因、影響、未解決の確認事項を PR に記載してください。

## 6. 完了報告・PR 記載

- 変更したファイルと目的を簡潔に記載してください。
- 実行した検証コマンドと結果を記載してください。
- 未実施の検証があれば、理由と影響を記載してください。
- Issue 外の判断、設計差異、既知の制約があれば明記してください。
- 追加で見つけた改善案は、今回の変更に混ぜず別 Issue 候補として扱ってください。
- 明示的な依頼なしに Issue を close しないでください。
- 明示的な依頼なしに PR を merge しないでください。
- 明示的な依頼なしに GitHub 設定、branch protection、Actions 設定を変更しないでください。
- 明示的な依頼なしに依存更新を行わないでください。
- PR 本文には対象 Issue への参照を含め、必要に応じて `Closes #...` を記載してください。
- 最終報告は、レビュー担当者が変更範囲と検証状況を追跡できる粒度にしてください。
